# 产品客服智能问答生成 — 设计文档

**日期**: 2026-05-30
**项目**: product-knowledge-base
**状态**: 待用户审阅

## 1. 背景与目标

为美客多（Mercado Livre 巴西站）在售产品，批量生成客服智能问答库，填入「多客（duoke）知识库模版」，供 AI 客服日常自动回复使用。

问答必须**基于真实数据**（真实买家历史问题 + 权威产品事实），不凭空编造，并产出**中文+葡语双语**（葡语供 AI 直接回复买家，中文供内部审核）。

## 2. 范围

- **目标产品**：表1（`WypHb12oZadMpps1sklcY27innC` / `tblPgP1nW2MW5FwH`）中的 **234 个在售 SKU**（仅为美客多上架产品生成）。
- 本次**不改动** `bot/` 运行时代码、不改动现有同步/导入流程；只新增一个独立的「问答生成」模块 + 输出 xlsx。

## 3. 关联键（已验证）

```
表1.SKU  ==  表2「库存sku编号」  ==  products.sku  ==  knowledge_entries.sku
例：CBC004-452
```

- 表1 提供：SKU 清单、美客多链接、店铺、运营。
- 表2（= `products` 表，`tbl1Zq6Sw6B5tP9x`，已同步进 Supabase，共 3722 条）提供产品基础信息与说明书。
- `knowledge_entries`（56,078 条，其中 customer_service 55,924 条，几乎全 approved）提供真实客服历史问答，按 SKU 关联。

## 4. 数据源（两条并列生成轨）

问答由**两个来源共同驱动**，对每个 SKU 都执行，最后合并去重。历史问题数据普遍不完善，**不能只靠历史**，必须同时从产品信息主动枚举生成。

1. **`knowledge_entries`（customer_service）— 历史问题轨**：该 SKU 的真实买家问题、意图、对话记录、客服「跟进建议」。提供「买家真实问过什么」和真实葡语问法。各 SKU 数量差异大（4 ~ 383 条），且普遍不完整。
2. **`products` — 产品信息轨（同为主驱动）**：`name_cn`、`features`、`description`（含尺寸/零件清单/参数）、`manual_files`（说明书链接），以及 `feishu_raw_data` 中的**基础信息白名单字段**（品牌/材质/用途/类目/认证类型与编号/包装材质/销售单位/装箱数等）。**既驱动问题生成**（系统化枚举产品属性，含丰富的基础信息类），**也作为答案事实来源**。⚠️ 成本/采购价/供应商/库存/开发员等**敏感字段一律不取**、不入答案。
3. **美客多页面（可选增强，firecrawl 抓取）**：葡语原文措辞、保修/物流条款。Phase 1 试跑时评估其增量价值，再决定批量是否保留。
4. **说明书 PDF 正文（可选）**：`extract_manual.py` 已存在但 Phase 2 因权限暂停；本期默认不依赖，若已有 `manual_content` 则纳入。

## 5. 生成逻辑（每个 SKU，两轨合并）

1. 拉取该 SKU 的全部 `knowledge_entries`（status=approved）+ `products` 事实 +（可选）ML 页面。
2. **轨 A — 历史问题挖掘**：把真实历史问题聚类去重为「不同的问题类型」（如：是否有某电压版本、产品故障排查、尺寸、配件清单、安装使用、保修退换、兼容性…），同一问题的不同表述合并。
3. **轨 B — 产品信息枚举**：系统化遍历 `products` 的 `features`/`description`/`name_cn`/说明书，按固定问题维度主动生成买家会关心的问题——无论历史是否问过。维度示例：
   - 规格尺寸、重量、颜色
   - 零件清单 / 包装含哪些
   - 核心参数（功率/电压/容量/材质等，依品类而定）
   - 功能用途、适用场景
   - 安装/使用方法、注意事项
   - 兼容性 / 电压制式 / 适配机型
   - 保修、退换、故障排查
   - 认证/安全（如有）
4. **合并去重**：轨 A + 轨 B 的问题取并集，去重（语义相近合并；真实问法优先保留）。
5. **生成问答**（LLM，复用项目现有 GPT-4o / DeepSeek 能力）：每个问题产出一条：
   - 葡语**问题标题**（简洁）
   - **6~10 种葡语问法**（优先取自真实历史表述，不足时由 LLM 扩展）
   - **葡语答案**（基于产品事实，完整、可直接回复；修正历史里客服答得不到位的点）
   - **中文参考答案**（葡语答案的中文对照，供审核）
6. **核查与一致性校验（第二遍 LLM，独立于生成）**：逐条把答案中的每个数字/事实回查三个来源，输出标记 `flag`：
   - **无任何来源支持** → 软化答案 + 标 `【待核实】`。
   - **ML 与内部数据冲突**（如内部持续 850W vs ML 1000W）→ 买家可见值以 ML 为准，并标 `【⚠️数据不一致 内部:X / ML:Y】`，方便人工核查（用户明确要求）。
   - **政策类问题**（运费/物流时效/退换货/保修）→ 替换为统一**通用模板**答案（不写具体天数/金额），标 `【通用模板·待补充政策】`。
   - 计数/数量（USB 口数、配件件数等）必须与来源逐字一致。
7. **数量**：不设上限。两轨合并后有多少有效问题就生成多少，尽可能覆盖产品信息中的全部有效内容。
8. **防编造原则**：答案只用已确认信息；严禁推算/换算；标记统一落在「中文参考」列，便于人工审核与补充。

## 6. 输出

- **格式**：8 列。前 3 列为导入用（葡语），后 5 列为审核用（导入多客前删除）：

  | # | 列 | 内容 | 用途 |
  |---|---|---|---|
  | 1 | 买家问题名称 | 葡语问题标题 | 导入 |
  | 2 | 买家可能问法 | 6~10 种葡语问法（换行分隔） | 导入 |
  | 3 | 答案 | 葡语答案（**多平台通用，不含任何平台名**） | 导入 |
  | 4 | 问题名称(中文) | q_pt 的中文翻译 | 审核 |
  | 5 | 可能问法(中文) | 各问法的中文翻译 | 审核 |
  | 6 | 答案(中文参考) | `[SKU]` 前缀 + 中文对照答案 | 审核 |
  | 7 | 数据来源 | 该答案信息来源（页面参数表/内部基础信息/客服历史/说明书…） | 审核 |
  | 8 | 核查标记 | `【待核实】`/`【⚠️数据不一致…】`/`【通用模板…】`，无则空 | 审核 |

- **多平台通用**：答案严禁出现具体平台名（Mercado Livre/Shopee 等），用中性表述。导入时仅保留前 3 列。

- **粒度**：一个 SKU 一个 xlsx 文件。
- **SKU 标注（便于检索）**：① 文件名带 SKU；② 每条「中文参考」列以 `[SKU]` 前缀开头（与 `knowledge_entries.title` 的现有约定一致）。葡语「问题名称/答案」保持干净，不加 SKU，避免影响 duoke 对买家消息的匹配。
- **命名**：`{SKU}_{产品简名}.xlsx`（产品简名取 name_cn 截断、去非法字符）。
- **输出目录**：`data/duoke_generated/`（项目内，纳入 .gitignore，不提交数据文件）。

## 7. 模块设计

新增脚本，复用 `scripts/utils.py`（Supabase 客户端、SKU 提取、语言检测）：

- `scripts/generate_product_qa.py` — 主流程：读表1 SKU 清单 → 逐 SKU 聚合数据 → 调 LLM 生成 → 写 xlsx。
  - 入参：`--sku CBC004-452`（单个，试跑）/ `--all`（批量）/ `--limit N`。
  - **默认 dry-run 友好**：不带显式参数不批量执行。
- 表1 读取：用 lark-mcp 凭据或项目飞书应用读取（实现时确认哪个应用有表1权限；表2/Supabase 已就绪）。
- LLM 提示词：内置「聚类去重 + 双语 + 防编造 + 多问法」规则；输入为该 SKU 的真实问答 + 产品事实。

## 7.1 更新与增量

源数据（新客服记录 / 产品信息更新）会持续变化，生成需可重复运行、支持增量：

- **清单文件** `data/duoke_generated/.manifest.json`：记录每个 SKU 上次生成时的源数据指纹——`knowledge_entries` 条数 + 最新 `created_at`、`products.updated_at`（或 feishu_raw_data 哈希）、生成时间。
- **增量模式**（`--incremental`）：只重跑指纹有变化的 SKU；无变化的跳过。
- **覆盖策略**：有变化的 SKU，其 xlsx **整体重新生成覆盖**。人工微调在 duoke 导入后进行，不在生成文件上手改（避免被覆盖丢失）。
- `--all` 强制全量重跑；`--sku` 指定单个。

## 8. 执行计划

- **Phase 1（试跑）**：`CBC004-452`（逆变器，8 条真实问答 + 丰富产品事实），含 ML 抓取。产出成品 xlsx 交用户审阅，确认质量/格式，并判断「ML 抓取是否值得在批量中保留」。

- **Phase 2（批量）**：用户确认后，批量处理表1 全部 234 个 SKU。

## 9. 约束与约定

- **不自动跑脚本**：所有脚本（含生成、同步、Supabase 读取）仅在用户明确同意时运行（遵循既有协作约定；并发同步曾致数据错乱）。
- Supabase 操作**只读**，不写 `products`/`knowledge_entries`。
- 不提交数据文件（xlsx/json）到 git。
- 提交信息遵循项目 CLAUDE.md 规范。

## 10. 未决项 / 风险

- **表1 读取应用权限**：实现时确认用哪个飞书应用读表1（lark-mcp 已验证可读）。
- **答案准确性**：依赖人工审核中文参考列；「【待核实】」项需运营补充。
