## 一、说明书提取（最大的坑，集中在这里）

### 1.1 飞书文件下载
- 说明书是飞书云空间文件（`…/file/{token}`），**必须用 `drive.v1.file.download`**，不是 `media.download`（后者对云文件返回空 → `response.file is None`）。
- 读取要兼容 `response.file` 与 **`response.raw.content`**（本 SDK 版本二进制在 raw.content）。
- 权限：应用需开 `drive:drive:readonly` / `drive:file:download` / `drive:file:readonly`（**已开通**，无需"多维表格附件下载"这种单独权限）。

### 1.2 文件格式识别
- 说明书文件名**可能没有扩展名**（如「二室一厅正开门双语说明书」）→ 不能只靠后缀判断。
- 已改为按 **magic bytes** 识别：`%PDF` → PDF；`PK` → docx/zip。

### 1.3 PDF 文字层不可靠 —— 两类都要 OCR
- **图片版 PDF（无文字层）**：如逆变器说明书，pypdf 抽出 0 字 → OCR。
- **CID 子集字体 PDF（最隐蔽）**：如帐篷说明书，pypdf 能抽出**10万+字，但全是 `/gid00010…` 字形乱码**，字数多容易被误判为"提取成功"。→ 必须检测乱码再 OCR。
  - 判定：含大量 `/gid` 或 `(cid:`、或字母/汉字占比 < 30% 视为乱码（`_looks_garbled`）。
- OCR 实现：PyMuPDF 渲染每页为图 → `tesseract -l chi_sim+por`。依赖：`pymupdf` + tesseract 的 `chi_sim`、`por` 语言包。默认只 OCR 前 20 页、dpi=200。
- **教训**：不能只用"文字长度 < 50"判断是否需要 OCR；乱码 PDF 长度很大但内容无效。

### 1.4 说明书是"配件清单/数量/安装步骤"的权威源
- 内部描述与说明书会冲突：帐篷内部写「14 根地钉」、说明书写「16 根」→ **以说明书为准**，并标 `【⚠️数据不一致 内部:14 / 说明书:16】`。
- 安装/使用/注意事项类问答**优先取说明书正文**（买家高频问"怎么装/怎么用/注意什么"）。

### 1.5 部分 SKU 没有说明书
- 如钓鱼轮 CBC004-342 的 `manual_files` 为空 `[]` → 跳过说明书，正常生成。

## 二、数据冲突与防编造

- **三源可能互相冲突**：ML 页面 / 内部 products / 说明书。优先级：买家可见规格(功率/尺寸/电压)以**页面**为准；配件清单/数量/安装以**说明书**为准；都没有再用内部。冲突取权威值并在中文参考标注差异。
- **LLM 实测会编造**，必须严防 + 标注：
  - 计数错误：内部「1 个 USB」→ 答「2 个」（**计数类 LLM 核查仍不可靠，批量需确定性正则校验或人工复核**）。
  - 凭空加配件：如"车载连接线"。
  - 推算/换算：由持续功率推峰值(850→1700)、"0.42 个地钉"等 → 严禁。
  - 政策默认值：自己编"30 天免费退货""满 R$19 免运费" → 政策类一律用**通用模板** + 标 `【通用模板·待补充政策】`。
- 无依据信息：保守作答，中文参考标 `【待核实】`。
- 每个 SKU 同时产出 `{SKU}_待核验.md` 列出需人工核验项。

## 三、美客多（ML）页面抓取

- **有反爬**：firecrawl 必须 `proxy: stealth`（basic 代理被登录墙挡，跳转 account-verification）。
- JSON 抽取模式不稳定（有时只回 metadata，且 stealth+json ~9 credits/次）→ 用 markdown + 锚点切片更稳。
- ML 页面输出巨大（~280K 字）→ **不要读进上下文**，按锚点（`O que você precisa saber` / `Características do produto` / `Descrição`）切片提取关键段。
- **无 ML 对照时核查更保守、`待核实` 条目暴增**（帐篷无ML时一度33条待核实）→ 批量尽量每 SKU 抓 ML 以减少人工复核量。

## 四、生成条数 / Token

- 单次生成调用受**输出 token 上限(~8000)**截断，条数受限，且会被说明书/ML 的输入上下文**挤占**（"加说明书后条数反而变少"就是这原因）。
- 解决：**多轮补充**（`--max-rounds`，每轮挖未覆盖问题，新增 < 5 停）+ JSON 截断**抢救解析**（salvage）。
- 翻译步**分批**（每 ~15 条一次）避免截断。

## 五、答案风格（多平台通用）

- **严禁出现任何平台名**（Mercado Livre / Shopee / 美客多 / Amazon…）；历史记录里的平台名一律忽略。
- **直接作答**：不甩"consulte a página do produto / verifique o anúncio"，不留来源引用尾巴（"conforme a página / ficha técnica / descrição"）——来源写进「数据来源」列。
- 兜底：`strip_citations` 正则清除引用尾巴。

## 六、输出与 SKU 标注

- xlsx **8 列**：前 3 列（葡语：问题/问法/答案）= 导入多客用；后 5 列（问题中文、问法中文、答案中文参考、数据来源、核查标记）= 审核用，**导入前删除后 5 列**。
- 文件名 `{SKU}_{产品名}.xlsx`；中文参考列带 `[SKU]` 前缀，便于检索。
- **敏感字段绝不入答案**：成本/采购价/供应商/库存/开发员/序号等。`feishu_raw_data` 只取白名单基础信息（品牌/材质/用途/类目/认证/包装/单位/装箱数等）。

## 八、批量运行 Checklist（234 个 SKU）

- [ ] **成本/限流**：每 SKU 约 = 生成 1 + 补充 3~4 + 核查 1 + 翻译 N 批（+ ML 抓取 1 + 说明书 OCR）次调用。234 个量级要算 DeepSeek/OpenAI 调用量与 firecrawl credits（stealth ~9/次）。做**并发限制 + 重试**。
- [ ] **断点续跑 / 增量**：用 `manifest.json` 记录每 SKU 源数据指纹，只重跑变化的；失败的可重试。
- [ ] **失败隔离**：单个 SKU 失败（下载/OCR/LLM）不中断整批，记录失败清单事后补。
- [ ] **OCR 耗时**：多页说明书逐页渲染 + tesseract 较慢，预留时间；限页数(默认20)。
- [ ] **抽检**：跑完按品类抽样人工核，**重点查计数类**（USB口数/配件件数）与 `【⚠️数据不一致】`、`【待核实】` 标注项。
- [ ] **无说明书 / 无 ML 的 SKU**：标识出来，其待核验条目会偏多，优先人工补。
- [ ] **不自动跑**：批量脚本经确认后再运行（既有协作约定，避免并发数据错乱）。

## 九、批量运行手册（防断 / 续跑 / 监控 / 中断恢复）

### 9.1 SKU 清单
- `data/duoke_generated/sku_list.json`（234 个，来自表1，含 sku/link/store）。由 lark-mcp 读表1生成（批量脚本本身不读表1）。

### 9.2 启动方式
```bash
cd ~/Projects/product-knowledge-base

# 验证波：只跑前 5 个，估时/查质量
/opt/homebrew/bin/python3.13 scripts/batch_generate_qa.py --limit 5 --max-rounds 2

# 全量（断点续跑，跳过 manifest 里已完成的）
/opt/homebrew/bin/python3.13 scripts/batch_generate_qa.py --max-rounds 2

# 指定/重跑
... --skus CBC004-452,CBC004-342      # 只跑这些
... --redo                            # 忽略 manifest 全部重跑
... --no-manual                       # 跳过说明书OCR（更快）
```

### 9.3 防断模式（推荐长跑）
脱离会话、防空闲休眠：
```bash
nohup caffeinate -i /opt/homebrew/bin/python3.13 scripts/batch_generate_qa.py --max-rounds 2 >> /tmp/batch_qa_full.log 2>&1 &
disown
```
- 切换/关闭 Claude 对话、退出 CLI **都不影响**它（已脱离会话）。
- `caffeinate -i` 防**空闲休眠**；但**合盖**（电池模式）可能仍睡 → 长跑时盖子别合 / 接电源。
- 关机也没关系：manifest 已存，重跑即续。

### 9.4 断点续跑（核心安全网）
- `data/duoke_generated/batch_manifest.json`：每个 SKU 完成/失败后**立即落盘**。
- 任何中断（Ctrl+C / 关机 / 进程被杀），直接重跑 9.2 全量命令即可**自动跳过已完成、从断点继续**。

### 9.5 ⚠️ 严禁并发
- **启动新批量前必须先杀掉旧进程**，否则两个进程并发处理同一批数据会冲突：
```bash
pkill -f batch_generate_qa.py; sleep 2   # 确认停了再重启
pgrep -fl batch_generate_qa.py            # 应为空
```

### 9.6 监控进度
- 脚本每完成 **10 个**打一条 `===PROGRESS===` 汇总到日志。
- 看实时进度：
```bash
tail -f /tmp/batch_qa_full.log | grep -E "===PROGRESS===|❌|批量结束"
# 或看 manifest 统计
/opt/homebrew/bin/python3.13 -c "import json;m=json.load(open('data/duoke_generated/batch_manifest.json'));import collections;print(collections.Counter(v['status'] for v in m.values()))"
```

### 9.7 完成后
- 输出在 `data/duoke_generated/`：每个 SKU 一个 `{SKU}_{名}.xlsx` + 一个 `{SKU}_待核验.md`。
- 失败清单见日志结尾「失败清单」或 manifest 里 `status=failed` 的项 → 排因后单独 `--skus` 重跑。
- 参考耗时：单 SKU ~195s（补充2轮），234 个全量 ≈ 12.6 小时（串行）。

## 十、第二批（运营一组 SKU）—— 按类目分两个文件夹

python3.13 scripts/batch_generate_qa.py --out-dir data/duoke_generated_CBC008 --max-rounds 2

**执行顺序**：接在第一批（CBC004，`data/duoke_generated/`）之后，链式自动跑：第一批结束 → BRME → CBC008，全程防断（caffeinate+nohup），日志 `/tmp/batch2_qa.log`。

- BRME/CBC008 若某 SKU 不在 Supabase `products` 表 → 记为失败（隔离不中断），跑完看各自 manifest 的 `failed`。
